# 第一部分 实验二：环境感知与交互控制实验

## 一、实验目的

1. 掌握 Hi3861 GPIO 外设的基本编程方法，包括 LED 灯控制和 ADC 按键检测
2. 理解 OpenHarmony LiteOS-M 嵌入式系统下"应用任务"的概念与启动流程
3. 掌握 IOT GPIO / HI ADC 等外设驱动的调用方式
4. 掌握硬件消抖（多次采样比较）的编程技巧
5. 学会将 LED 输出与按键输入组合，实现简单的"按键控制 LED"交互逻辑

| 烧录工具 | HiBurn (Windows) 115200bps |

## 三、实验原理

### 3.1 GPIO 基础

Hi3861 提供丰富的 GPIO 资源。在 OpenHarmony 标准的 IoT 外设接口中，使用如下 API：

```c
IoTGpioInit(GPIO_NUM);              // 初始化 GPIO
IoTGpioSetDir(GPIO_NUM, DIR);       // 设置方向（输入/输出）
IoTGpioSetOutputVal(GPIO_NUM, VAL); // 输出高/低电平
IoTGpioGetInputVal(GPIO_NUM, &val); // 读取输入电平
```

### 3.2 ADC 采样

Hi3861 的 ADC 精度为 12 bit（0~4095），参考电压 1.8V，外加 4× 模拟增益。本实验的 ADC3 通道用于采集按键电路的模拟电压：

```c
hi_adc_read(HI_ADC_CHANNEL_3, &data,
            HI_ADC_EQU_MODEL_4,      // 4 次平均
            HI_ADC_CUR_BAIS_DEFAULT, // 默认偏置
            0);                      // 不用额外延时
```

由于 ADC 采样存在抖动，本实验使用**软件消抖**：连续两次采样值接近、且都低于阈值，才认定为按下。

### 3.3 LiteOS-M 任务模型

LiteOS-M 任务入口函数 `void *task(void *arg)` 形如 pthread 风格；通过 `osThreadNew()` 创建。`APP_FEATURE_INIT` 宏让开发者只写一个 `xxx_Entry()`，由系统初始化器在合适时机会调用。

## 四、代码说明

### 4.1 LED 闪烁（Lab4 子实验 4.1）

源码：`Lab4/key_demo/key_demo.c`（同时实现 LED 闪烁与 ADC 按键检测）

```c
void *KeyTask(const char *arg)
{
    (void)arg;
    IoTGpioInit(LED_GPIO);
    IoTGpioSetDir(LED_GPIO, IOT_GPIO_DIR_OUT);

    int last_pressed = 0;
    while (1) {
        int pressed = IsKeyPressed();
        if (pressed && !last_pressed) {
            unsigned int val = ReadAdcKey();
            if (val < 400) {
                led_set(1);  // KEY1: 点亮 LED
            } else if (val < 800) {
                led_set(0);  // KEY2: 熄灭 LED
            }
        }
        last_pressed = pressed;
        usleep(50000);  // 50ms 扫描
    }
}
```

### 4.2 ADC 按键消抖

源码同上文件，关键代码：

```c
static int IsKeyPressed(void)
{
    unsigned int val1 = ReadAdcKey();
    usleep(10000);                 // 10ms 后再次采样
    unsigned int val2 = ReadAdcKey();
    if (val1 < 2000 && val2 < 2000) return 1;  // 两次都低，确认为按下
    return 0;
}
```

### 4.3 "按键 → LED"交互

KEY1 点亮 LED，KEY2 熄灭 LED。通过 `IoTGpioSetOutputVal(GPIO9, IOT_GPIO_VAL_LOW/HIGH)` 实现控制电平翻转。

## 三、实验原理

### 3.1 WiFi STA 连接流程

```c
hi_wifi_init(0);                                  // 初始化 WiFi 协议栈
hi_wifi_start_sta();                              // 启动 STA 模式
hi_wifi_register_event_callback(wifi_event_cb);   // 注册事件回调
hi_wifi_sta_connect(&assoc_req);                 // 发起连接
netifapi_dhcp_start(g_lwip_netif);                // 启动 DHCP 客户端
```

事件回调在另一线程中执行，通过 `g_wifi_connected` 全局标志位通知主任务。

### 3.2 UDP 服务器

UDP 套接字创建后，通过 `bind()` 绑定本地 IP+端口，进入 `recvfrom()` 循环：

```c
sockfd = socket(PF_INET, SOCK_DGRAM, 0);
bind(sockfd, (struct sockaddr *)&servaddr, sizeof(servaddr));
while (1) {
    recvfrom(sockfd, buf, sizeof(buf), 0,
             (struct sockaddr*)&client, &len);
    // 解析 JSON 命令
    // 发送响应
    sendto(sockfd, response, len, 0,
           (struct sockaddr*)&client, sizeof(client));
}
```

### 3.3 TCP 客户端

TCP 客户端主动 `connect()` 服务器，三次握手成功后即可 `send()` / `recv()`：

```c
sockfd = socket(PF_INET, SOCK_STREAM, 0);
connect(sockfd, (struct sockaddr*)&server, sizeof(server));
send(sockfd, "hello", 5, 0);
recv(sockfd, buf, sizeof(buf), 0);
```

本实验让 Hi3861 作为 TCP 客户端连接到 PC 上的 TCP 服务器，**每 5 秒发送一次心跳**（JSON 格式），同时被动接收服务器下发的命令。

## 四、代码说明

### 4.1 UDP 服务器

源码：`Lab3/udp_server/udp_server.c`

**核心结构：**

| 阶段 | 关键调用 |
|:-----|:---------|
| WiFi 连接 | `wifi_init_and_connect()` |
| 套接字创建 | `socket(PF_INET, SOCK_DGRAM, 0)` |
| 地址复用 | `setsockopt(..., SO_REUSEADDR, ...)` |

**支持的 JSON 命令：**

```json
{"cmd": "forward"}   → 前进
{"cmd": "backward"}  → 后退
{"cmd": "left"}      → 左转
{"cmd": "right"}     → 右转
{"cmd": "stop"}      → 停止
{"cmd": "query"}     → 查询
{"mode": "step"}     → 步进
{"mode": "alway"}    → 持续
```

每条命令都通过 `sendto()` 回送 `{"result":"ok"}`。

### 4.2 TCP 客户端

源码：`Lab3/tcp_client/tcp_client.c`

**核心特性：**

1. 启动时发送注册包：`{"device":"hi3861","status":"online"}`
2. 每 5 秒发送心跳：`{"type":"heartbeat","time":0}`
3. 接收超时 100ms，检测对端关闭
4. 失败自动重连，最多 10 次

**重连机制：**

```c
connect_retry:
    sockfd = socket(...);
    ret = connect(sockfd, ...);
    if (ret < 0) {
        if (retry_count++ < 10) {
            osDelay(30);     // 3 秒后重试
            goto connect_retry;
        }
    }
```

## 三、实验原理

### 3.1 MQTT 协议核心概念

| 概念 | 含义 |
|:-----|:-----|
| Broker | 消息代理，本实验使用 mosquitto |
| Topic | 主题，消息分类路径，如 `ohospub` / `ohossub` |
| Publisher | 发布者，发送消息到某 Topic |
| Subscriber | 订阅者，接收某 Topic 消息 |
| QoS | 0=最多一次, 1=至少一次, 2=恰好一次 |

### 3.2 Paho MQTT API（LiteOS 移植版）

```c
NetworkInit(&n);
NetworkConnect(&n, "192.168.43.100", 1883);
MQTTClientInit(&client, &n, 3000, send_buf, send_size, recv_buf, recv_size);
MQTTPacket_connectData d = MQTTPacket_connectData_initializer;
d.clientID.cstring = "Hi3861_Client";
MQTTConnect(&client, &d);
MQTTSubscribe(&client, "ohossub", QOS1, message_arrived);
MQTTPublish(&client, "ohospub", &msg);
MQTTStartTask(&client);   // 启动接收任务
```

### 3.3 mosquitto 启动

`mosquitto.conf` 关键配置：

### 4.3 接收回调（消息解析）

```c
void mqtt_arrived(MessageData *msg_data) {
    // 拷贝 payload
    int len = msg_data->message->payloadlen;
    char *msg = malloc(len + 1);
    memcpy(msg, msg_data->message->payload, len);
    msg[len] = '\0';

    // 简单 JSON 子串匹配
    if (strstr(msg, "led_on"))      led_set(1);
    else if (strstr(msg, "led_off")) led_set(0);
    else if (strstr(msg, "toggle")) led_set(!led_get());
    free(msg);
}
```

# 第四部分 实验五：设备上云（华为云 IoTDA）

## 一、实验目的

1. 掌握华为云 IoTDA 平台的设备接入流程
2. 理解端云协同：数据上报（属性）+ 命令下发（服务调用）
3. 掌握 MQ-2 气体传感器的硬件接口与 ADC 采样
4. 学会在嵌入式端使用 cJSON 库构造 / 解析 JSON
5. 实现"异常检测 → 告警闪烁 → 自动上报"的智能场景联动

## 二、实验环境

| 项目 | 说明 |
|:-----|:-----|
| 开发板 | HiSpark Pegasus + 环境监测扩展板 |
| 传感器 | MQ-2 可燃气体传感器（ADC5 通道） |
| 云平台 | 华为云 IoTDA 标准版（cn-north-4） |
| 协议 | MQTT over TCP（端口 1883） |
| 依赖 | Paho MQTT, cJSON, easy_wifi |

## 三、实验原理

### 3.1 华为云 IoTDA 标准 Topic

```
$oc/devices/{device_id}/sys/properties/report   ← 属性上报
$oc/devices/{device_id}/sys/commands/#          ← 命令订阅
$oc/devices/{device_id}/sys/commands/response   ← 命令响应
```

### 3.2 设备三元组

| 字段 | 说明 |
|:-----|:-----|
| `DEVICE_ID` | 设备 ID |
| `MQTT_CLIENT_ID` | 格式 `{device_id}_0_0_{timestamp}` |
| `MQTT_USERNAME` | 通常等于 `device_id` |
| `MQTT_PASSWORD` | HMAC-SHA256(device_secret, "deviceId{id}timestamp{ts}clientId{cid}versionv3.0") |

### 3.3 物模型（Profile）

```json
{
  "service_id": "GasValue",
  "properties": {
    "gas_value": "12.34"
  }
}
```

### 3.4 MQ-2 电阻计算

```
           Vcc (5V)
            │
            ├──[ MQ-2 传感器 Rx ]──┬── ADC5 采样点
                                  │
                              [ 1kΩ ]
                                  │
                                 GND

分压公式: Vx / 5V = 1kΩ / (1kΩ + Rx)
                => Rx = 5 / Vx − 1  (kΩ)
```

浓度越高 → 传感器电阻 Rx 越小 → 分压 Vx 越大 → ADC 数值越大。

float GetGasLevel(void) {
    unsigned short raw_adc = 0;
    hi_adc_read(HI_ADC_CHANNEL_5, &raw_adc,
                HI_ADC_EQU_MODEL_4, HI_ADC_CUR_BAIS_DEFAULT, 0);
    // 1.8V 参考 × 4 增益 / 4095 满量程
    float Vx = (float)raw_adc * 1.8f * 4.0f / 4096.0f;
    float Rx = 0.0f;
    if (Vx > 0.01f) {

    return Rx;
}

# 第五部分 心得总结

四个实验的难度阶梯很清晰，所以我顺着**从"调通一行 `printf`"到"调通一条端到端链路"**的过程，整理出几条真正让我产生印象、并且现在能讲清楚"为什么这样做"的具体体会。

### 一、嵌入式开发里"看似正确但实际不可靠"才是常态

实验二第一次做 ADC 按键检测时，我写了一个判断"`< 2000` 就认为按下"的版本，单次采样通过，但上板之后 50 次里大概有 5~8 次会被毛刺触发。原因很简单：Hi3861 的 ADC 没有任何硬件消抖，按键簧片在 5~10 ms 内的回弹会被采到 1~2 次异常值。

`key_demo.c` 里最终的实现是**两次采样 + 10 ms 间隔 + 阈值比较**：

```c
unsigned int val1 = ReadAdcKey();
usleep(10000);
unsigned int val2 = ReadAdcKey();
if (val1 < 2000 && val2 < 2000) return 1;
```

50 ms 扫描一次的话，一次按键大致能采到 30~40 个连续低值，最终报告的"按下"事件非常稳定。这个细节在 PC 程序里几乎遇不到，但它直接决定了外设能不能"用"。

### 二、UDP 和 TCP 选型要看"丢一次能不能接受"

实验三的两个子任务不是随便切的。我先做了 `udp_server.c`：Hi3861 监听 50001，`recvfrom` + `sendto` 收发 JSON 命令，每条命令立刻回 `{"result":"ok"}`。这种"一发一回"用 UDP 很自然，因为丢一帧命令的后果是用户再点一次按钮。

但 `tcp_client.c` 走的却是 TCP，理由也很直接——**心跳包必须可靠**。我需要服务器在 5 秒内看到一次 `{"type":"heartbeat"}`，丢一个心跳会触发对端误判离线。`tcp_client.c` 里我把 `SO_RCVTIMEO` 设到 100 ms 兼做"对端关闭"探测；`connect` 失败后用 `goto connect_retry` 限制最多 10 次重试，避免开发板在断网后死循环刷日志、烧 flash。

`tcp_client.c` 第 154~166 行这段重连逻辑看着粗暴，但在 352 KB SRAM 的设备上，比定时器队列更可控。

### 三、MQTT 让我重新理解了"多对多"

在做实验四之前，我对发布/订阅的印象还停留在"一个推一个收"。`smart_home.c` 把三个并发任务拼在一起，才让我真正看到 Paho 的模型是怎样工作的：

```
WiFi 任务  ──┐
按键扫描 ────┼──► 共享 g_led_state ──► MQTT 任务 ──► smart_home/hi3861/status
MQTT 回调 ──┘                              ▲
                                           │
                                  smart_home/led/control
```

PC 上的 MQTTX 既能**发布** `{"cmd":"led_on"}` 控制 LED，也能**订阅** `smart_home/hi3861/status` 看到设备 20 秒一次的心跳。设备和 PC 的角色完全对称——**谁订阅谁就能收，谁发布谁就能收**，设备并不需要知道对端是谁。

我做 `mqtt_local.c` 时为了"消息真的有回环"，特地开了两个 MQTTX 窗口：一个订阅 `ohospub`，一个发布到 `ohossub`，分别验证了上行（设备→Broker→订阅者）和下行（订阅者→Broker→设备回调 `mqtt_message_arrived()`）两条路径。

### 四、华为云 IoTDA 把"协议一致性"变成了产品要求

实验五踩的坑最多，因为华为云的物模型对 JSON 格式非常严格，差一个字段或一个反斜杠都会被设备影子拒收。

第一版 `main.c` 上报时串口打印 `Report #0` 一切正常，但云端影子一直是空值。检查后发现是两层问题：

1. **设备命令的字段名**：必须用 `command_name` / `paras` / `result_code` / `response_name`，不能自己造（比如 `cmd_name` 不会触发响应）。
2. **`request_id` 必须从 topic 中提取并回传**：华为云下发的命令 topic 形如 `$oc/devices/{id}/sys/commands/request_id=abc123/...`，响应必须发到 `$oc/devices/{id}/sys/commands/response/request_id=abc123`，否则云端会一直显示"未响应"。

```c
char *req_id_ptr = strstr(topic, "request_id=");
if (req_id_ptr) {
    char *start = req_id_ptr + 11;
    int i;
    for (i = 0; i < 36 && start[i] && start[i] != '/'; i++)
        request_id[i] = start[i];
    request_id[i] = '\0';
}
snprintf(resp_topic, sizeof(resp_topic),
         "%s/request_id=%s", RESPONSE_TOPIC, request_id);
```

改完之后，云端"消息跟踪"里立刻能看到绿勾。这件事让我意识到，云端协议不是"按文档做就行"，而是要把"topic 里的元数据当作一等公民"。

### 五、智能场景联动里"上报警状态"比"上报数值"更重要

`main_v2_smart_scene.c` 增加的 `auto_control_task` 在功能上就是"每 200 ms 读一次 ADC、判断阈值、推一次消息"。但如果每次都把当前 `gas_value` 推上去，10 秒一次主循环已经够了，独立任务的真正价值在于**状态变化的瞬时事件**：

```c
if (!g_warning_active && should_warn) {
    g_warning_active = true;
    MQTTPublish(&g_mqtt_client, PUBLISH_TOPIC, &msg);  // QoS1 立即推一次
}
```

阈值我分了 30 kΩ / 15 kΩ / 8 kΩ 三档，分别对应 `warning` / `danger` / `critical`：

| 阈值 (kΩ) | 等级 | 上报字段 | 设备动作 |
|:---------:|:-----|:--------|:---------|
| Rx ≥ 30 | normal | `alert_level=normal` | 静默上报 |
| 15 ≤ Rx < 30 | warning | `alert_level=warning` | 10s 周期闪烁 |
| 8 ≤ Rx < 15 | danger | `alert_level=danger` | 2s 快速闪烁 |
| Rx < 8 | critical | `alert_level=critical` | 1s 急促闪烁 |

按 MQ-2 数据手册的典型值，空气中 Rx 应在 25~40 kΩ 区间，遇到打火机泄漏的可燃气体后能掉到 12~15 kΩ；如果串口能看到连续 `Report #N | Gas: 12.34 kΩ | Alert: ALARM` 这种行，就说明 `auto_control_task` 被正确触发了。**注：这份数据基于 MQ-2 手册与公开实验报告，提交时实际串口输出未单独拍照归档，请按 Lab5 README 步骤复现验证。**

### 六、Hi3861 的 352 KB SRAM 教会我做减法

写 PC 程序我很少考虑 `malloc`，但在 Hi3861 上每次分配都心疼。`mqtt_local.c` 里我开了 4 KB × 2 的 MQTT 收发缓冲区：

```c
int buf_size = 1024 * 4;
g_send_buf = (unsigned char *)malloc(buf_size);
g_recv_buf = (unsigned char *)malloc(buf_size);
```

4 KB 是 Paho 的下限要求，再小就装不下 QoS2 的报文。`main.c` 里干脆用**栈上静态数组** `g_send_buf[2000]`，省掉了 4 KB 堆内存——这是 `mqtt_gas_sensor` 跑得比 `mqtt_local` 更稳的一个隐性原因。

cJSON 同样要小心：`cJSON_PrintUnformatted` 返回的字符串必须用 `cJSON_free` 释放，根节点必须 `cJSON_Delete`。我专门为每个错误分支写了对应的 `cJSON_Delete` 路径，避免在 `properties` 失败时只释放 `root` 造成 service 节点泄漏。这种"显式资源管理"在 PC 上是良好习惯，在嵌入式上是必要条件。

### 七、这次实验的"副产品"

`web_control_panel.py` 是在做 Lab5 之后加的：既然 MQTTX 只能看一个主题，我就用 Flask + paho-mqtt 自己写了一个面板，把设备状态、燃气值、LED 状态、历史曲线都画出来。开发板还是那个开发板，多了一层呈现方式，调试时肉眼就能看到告警等级颜色变化（绿/橙/红）。

这次实验前我以为 OpenHarmony LiteOS-M 只是一个"功能比 Arduino 强一点"的开发框架，做完之后意识到它已经把**任务调度、消息队列、WiFi、TCP/IP、MQTT 协议栈**封装成可拼装的组件——只要在 `BUILD.gn` 里写好 `deps`，编译器就把所有依赖接上，省下了所有"自己移植 lwIP"的工作量。下一步我想试试 LiteOS-A + 标准系统，对比一下同一套 OpenHarmony 体系下，不同形态的设备能做到什么程度。
